# AI Studio Architecture

Version: 1.0  
Status: Technical Architecture

---

# Purpose

This document describes the current technical architecture of AI Studio.

It explains how the application is organized, how components interact, and the architectural principles followed throughout the project.

This document reflects the current implementation only.

Future designs and planned features are intentionally excluded.

---

# Architectural Principles

The architecture is designed around the following principles:

- Local-first
- Modular
- Service-oriented
- Incremental development
- Filesystem-based persistence
- Minimal dependencies
- Production-quality code
- Maintainability over complexity

---

# Request Flow

All user interactions follow the same general flow.

```text
Browser

↓

FastAPI Route

↓

Service

↓

Filesystem

↓

Response

↓

Browser
```

Business logic belongs in services.

Routes remain lightweight.

---

# Service Layer

Services are responsible for implementing application behavior.

General responsibilities include:

- Reading data
- Writing data
- Validation
- Business logic
- File management

Services should not contain presentation logic.

---

# Routing Layer

Routes are responsible for:

- Receiving requests
- Validating input
- Calling services
- Returning responses

Routes should remain lightweight.

Business logic belongs in services.

---

# Storage

AI Studio uses the local filesystem for persistence.

Current design:

- No database
- No cloud storage
- No external persistence layer

Application data is stored within the workspace directory.

---

# Story Pipeline

Current workflow:

```text
Story

↓

Expanded Story

↓

Scenes
```

Each stage is stored independently.

This allows users to modify each stage without affecting previous work.

---

# AI Integration

AI functionality is provided through LM Studio.

The application communicates using the OpenAI-compatible API.

The implementation is designed so that local models can be changed without affecting the overall application architecture.

---

# User Interface

The web interface is built using:

- FastAPI
- HTMX
- Jinja2
- Minimal JavaScript

Rendering is primarily server-side.

---

# Testing

Testing is performed using pytest.

Testing focuses on:

- Services
- Routes
- Persistence
- Project management
- Story workflow

Manual verification is performed for user-facing functionality when appropriate.

---

# Design Rules

The following architectural rules apply throughout the project.

- Preserve existing architecture.
- Prefer extending existing services.
- Keep services focused.
- Keep routes lightweight.
- Avoid unnecessary abstractions.
- Use local filesystem persistence.
- Maintain backward compatibility where practical.
- Keep modules independent.
- Keep functions readable.
- Avoid unnecessary refactoring.
